## 1. 문제 해결 전략: 메모이제이션(Memoization)과 시간 복잡도

피보나치 수열의 기본 점화식 `f(n) = f(n-1) + f(n-2)`을 단순 재귀로 구현할 경우, 시간이 아주x100 오래걸린다.

### 1.1. 단순 재귀의 문제점: 중복 부분 문제 (Overlapping Subproblems)

단순 재귀를 사용하면 한 번 계산했던 값을 기억하지 못하고 똑같은 연산을 수없이 반복하게 된다.

예를 들어 5번째 수를 구하기 위해 3번째 수를 두 번 계산해야 하며, N이 커질수록 이 중복 연산은 기하급수적으로 늘어난다. 단순 재귀 방식의 시간 복잡도는 O(2^N)으로, N=200일 경우 시간 초과(Time Limit Exceeded)가 발생한다.

### 1.2.  메모이제이션

메모이제이션은 한 번 계산한 결과는 메모(캐시)해 두고, 다시 필요할 때 재사용하는 기법이다.

- 전역에 `memo`라는 배열을 선언하여 빈 노트를 준비한다.
- 특정 N에 대한 계산이 완료되면 그 결과를 `memo[N]`에 저장한다.
- 이후 동일한 N에 대한 함수 호출이 발생하면, 하위 재귀로 내려가지 않고 `memo[N]`에 저장된 값을 즉시 반환(Return)한다.

이 기법을 적용하면 수많은 연산 가지들이 쳐내지면서 시간 복잡도가 O(N)으로 급격히 단축된다. 즉, N=200이라도 200번의 기본 연산만으로  정답을 도출할 수 있다.

## 3. 코드 동작 과정 (ex. N=200)

작성된 메모이제이션 코드에 N=200이 입력되었을 때, 프로그램은 Top-down과 Bottom-up을 거친다.

1. **Top-down: 스택 쌓기**
    - `main` 메서드가 `fibonacci(200)`을 호출한다.
    - `fibonacci(200)`은 값을 구하기 위해 `fibonacci(199)`를 호출하고 대기한다.
    - `fibonacci(199)`는 `fibonacci(198)`을 호출하고 대기한다.
    - 이 과정이 반복되어 N이 `2`와 `1`이 될 때까지 계속 함수를 파고들어 간다.
2. **Base Case 도달 (Return 시작)**
    - 마침내 `fibonacci(2)`와 `fibonacci(1)`에 도달하면, 조건문 `(n == 1 || n == 2)`에 의해 각각 **1**을 즉시 반환(Return)한다.
3. **Bottom-up & 메모이제이션** 
    - 대기 중이던 `fibonacci(3)`은 반환받은 두 값(1+1)을 더해 **2**를 구한다. 이 값을 `memo[3] = 2`에 저장하고 상위로 반환한다.
    - 다음으로 `fibonacci(4)`는 반환받은 `fibonacci(3)`의 값(2)과, `fibonacci(2)`의 값(1)을 더해 **3**을 만들고 `memo[4] = 3`에 저장한다.
    - 이렇게 순차적으로 `memo[5]`, `memo[6]`... 에 값들이 차례대로 기록되며 200층을 거슬러 올라온다.
4. **메모이제이션과 일반 재귀의 차이점**
    - 연산 중간에 `fibonacci(200)`이 `fibonacci(199)`의 값을 받고 나서, 나머지 절반인 `fibonacci(198)`을 호출하는 시점을 생각해보자.
    - 일반 재귀라면 `fibonacci(198)`을 구하기 위해 다시 197, 196... 밑바닥까지 내려가야 함.
    - 하지만 이 코드에서는 `fibonacci(199)`를 구하는 과정에서 이미 `memo[198]`에 값이 저장되어 있음 →   ****`(198)`이 호출되면  `if (memo[n] != 0)` 조건문에 걸려 **연산을 하지 않고 배열에 저장된 값을 즉시 반환**한다.
    
5. **최종 계산 및 종료**
    - 수많은 가지치기 끝에 `fibonacci(200)`은 저장된 `memo[199]`와 `memo[198]`을 더해 (10009로 나눈 나머지를) 최종 값으로 산출한다.
    - 이 값을 반환하면 `main` 메서드에서 콘솔에 출력하고 프로그램이 안전하게 종료된다.

## ++) 다른방법: 꼬리 재귀(Tail Recursion) 방식 & 폐기된 이유

메모이제이션 배열을 사용하지 않고 변수만을 활용하여 연산 속도를 높이는 **꼬리 재귀**라는 방법도 존재한다. 메모이제이션이 너무 뻔한? 해결방법인 것 같아서 이걸로도 해봤는데,   클로드가 이 코드를 보고는 숫자가 더 커지면 스택오버플로우 난다고 폐기시킴 . 자바(Java) 환경에서는 이 방식을 쓸 수가 없었음

### 2.1. 꼬리 재귀란?

결과값을 함수의 매개변수(Parameter)로 계속 넘겨주며 계산을 이어가는 방식이다.

- 예: `fibonacci(n - 1, 현재 값, 이전 값 + 현재 값)`
- 계산이 완료된 값을 들고 다음 단계로 넘어가기 때문에, 이전 단계로 다시 되돌아와서 추가 연산을 할 필요가 없다. 시간 복잡도는 메모이제이션과 동일한 O(N)이다.

```java
import java.util.Scanner;

public class Main {
public static void main(String[] args) {
Scanner sc = new Scanner([System.in](http://system.in/));
   
    if (sc.hasNextInt()) {
        int n = sc.nextInt();
        // n, 1번째 수(1), 2번째 수(1)를 매개변수로 전달하며 시작
        System.out.println(fibonacci(n, 1, 1));
    }

    sc.close();
}

// n: 남은 단계, a: 이전 값, b: 현재 값
public static int fibonacci(int n, int a, int b) {
    if (n == 1) return a;
    if (n == 2) return b;

    // 다음 단계로 갈 때, 이전 값(a)과 현재 값(b)을 더해서 매개변수로 넘긴다.
    return fibonacci(n - 1, b, (a + b) % 10009);
}
}
```

### 2.2. 자바(Java) 메모리 구조로 인한 한계

프로그램이 함수를 호출하면, 컴퓨터는 메모리의 '콜 스택(Call Stack)'이라는 공간에 해당 함수의 실행 정보(변수, 돌아갈 위치 등)를 담은 프레임을 생성한다. 재귀 함수가 호출될 때마다 이 스택 프레임은 위로 차곡차곡 쌓이게 된다.

- **꼬리 재귀 최적화(TCO)의 부재:** C++이나 Python의 일부 환경에서는 컴파일러가 꼬리 재귀를 감지하면, 스택을 새로 쌓지 않고 기존 스택을 덮어쓰는 형태(마치 `for`문처럼)로 내부 최적화를 진행한다.
- **StackOverflowError:** 반면 자바의 가상 머신(JVM)은 이 꼬리 재귀 최적화를 지원하지 않는다. 따라서 꼬리 재귀를 쓰더라도 재귀가 깊어지면 정직하게 스택 프레임을 계속 쌓아 올린다.
- **결론:** 이 문제에서는 N=200으로 제한되어 있어 스택이 200층까지만 쌓이므로 무사히 실행된다. 하지만 만약 N이 100,000 등으로 커진다면, 메모리 한도를 초과하여 프로그램이 강제 종료되는 `StackOverflowError`가 무조건 발생한다.
